1 Overview

This tool deploys a Cloudflare Worker in your own account that reads your existing Logpush R2 logs, computes per-minute origin pull bandwidth using the official Cloudflare formula, and presents the results as a real-time chart dashboard.

ItemDetails
PurposeVisualize how much bandwidth your origin server is serving to Cloudflare (i.e., cache miss traffic)
Stack100% Cloudflare-native: Logpush + R2 + Queues + Workers + D1
DeploymentDeployed in your own Cloudflare account — no cross-account access, no external servers
R2 AccessRead-only via R2 Binding — never writes to your R2
Data Retention7 days (configurable), auto-cleaned daily
Chart Lag~10–15 minutes (Lag Window 7 min + Cron retry ≤8 min + processing ~1 min)

Key Design Decisions

Streaming Gzip Decompression: Logpush files are processed via DecompressionStream piped from the R2 object stream. The Worker decompresses and parses line-by-line, keeping memory constant regardless of file size. This is essential given the 128 MB Worker memory limit.

Idempotent Accumulative Writes: D1 writes use INSERT ... ON CONFLICT DO UPDATE SET sum_bytes = sum_bytes + excluded.sum_bytes. Multiple files covering the same minute accumulate correctly. If a file is retried, the processed_files status check prevents double-counting.

R2 Event Notification as Primary Trigger: Rather than polling R2 every N minutes, we rely on R2 Event Notifications for near-real-time processing. A secondary Cron (*/8 * * * *) retries files that were deferred due to the Safe Lag Window.

Safe Lag Window: Files newer than now - LAG_MINUTES are skipped. This ensures that all Logpush files covering a given minute have been written to R2 before we start aggregating that minute. Without this, the last few minutes of data would be incomplete.

3 Formula

Official Cloudflare formula for computing bytes served by the origin:

SUM(CacheResponseBytes)
WHERE OriginResponseStatus NOT IN (0, 304)
÷ 60 seconds × 8 bits = Mbps

Reference: developers.cloudflare.com/logs/faq/common-calculations

OriginResponseStatusMeaningIncluded in bandwidth?
0HIT — CF served from cache, no request made to origin❌ Excluded
304Revalidated — origin confirmed no change, no body transferred❌ Excluded
200, 206Origin served content (full or partial). Main source of origin bandwidth.✅ Included
4xx, 5xxOrigin error responses (included per official formula)✅ Included

Time bucketing: uses EdgeEndTimestamp (the moment CF finishes sending the response to the client), consistent with the CF Analytics dashboard. For large file downloads that span multiple minutes, this correctly assigns the traffic to when it completed.

4 Deployment

Prerequisites: Node.js v18+, Wrangler CLI v3+ (npm install -g wrangler), a Cloudflare account with Workers Paid plan (required for D1 + Queues), and an existing Logpush job writing to R2.

Step 1 — Create D1 Database

wrangler d1 create bw-stats

Copy the database_id from the output and paste it into wrangler.toml:

[[d1_databases]]
binding       = "DB"
database_name = "bw-stats"
database_id   = "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"   # ← paste here

Step 2 — Initialize D1 Tables

wrangler d1 execute bw-stats --file=./schema.sql --remote

Verify:

wrangler d1 execute bw-stats \
  --command="SELECT name FROM sqlite_master WHERE type='table'" --remote

Expected output: bw_stats and processed_files.

Step 3 — Create Queues

wrangler queues create bw-ingest-queue
wrangler queues create bw-ingest-dlq

Step 4 — Edit wrangler.toml

Open wrangler.toml and fill in your values:

[[r2_buckets]]
binding     = "LOGPUSH_BUCKET"
bucket_name = "your-logpush-bucket"   # ← your actual R2 bucket name

[[d1_databases]]
database_id = "xxxxxxxx-..."          # ← from Step 1

[vars]
LOG_PREFIX = "logs/"                  # ← Logpush path prefix (usually "logs/")

Step 5 — Configure R2 Event Notification

This is the primary trigger that enables near-real-time processing.

  1. CF Dashboard → R2 → your Logpush bucket → Settings → Event Notifications
  2. Click Add notification
  3. Set: Event type = object-create, Queue = bw-ingest-queue
  4. Optional: Prefix = logs/, Suffix = .gz
Important: Without R2 Event Notification, the Worker only runs on the 10-minute Cron schedule. Event Notification is strongly recommended for timely processing.

Step 6 — Set Dashboard Token (Optional)

# Leave empty to disable authentication (development only)
# For production, set a strong random string:
wrangler secret put DASHBOARD_TOKEN

Step 7 — Deploy

wrangler deploy

Expected output confirms: D1 binding, R2 binding, Queue producer/consumer, two Cron schedules.

Step 8 — Verify

# Check system status
curl "https://origin-bw-monitor.<subdomain>.workers.dev/api/status"

# Open dashboard
# https://origin-bw-monitor.<subdomain>.workers.dev/?zone=example.com&hours=24

5 Configuration Reference

wrangler.toml [vars]

VariableDefaultRequiredDescription
LOG_PREFIXlogs/RequiredPath prefix inside R2 for Logpush files. Must match your Logpush job configuration.
LAG_MINUTES7OptionalSafe Lag Window in minutes. Files newer than now - LAG_MINUTES are skipped to ensure all Logpush files for a given minute have landed in R2. Do not set below 5.
SCAN_BATCH_SIZE20OptionalMax files to enqueue per Cron run. Controls max files retried per Cron run.
MAX_PROCESSING_MIN15OptionalIf a file stays in processing status for longer than this (minutes), it is assumed the Worker crashed and the file is re-queued.
DATA_RETENTION_DAYS7OptionalD1 data retention in days. Older data is deleted daily at UTC 02:00 by the cleanup Cron.
LOG_LEVELinfoOptionalWorker log verbosity: debug / info / warn / error. Use debug for troubleshooting; revert to info in production.
DASHBOARD_TOKENemptyOptionalAccess token for the Dashboard UI. Empty = no authentication (development only). Production: set via wrangler secret put DASHBOARD_TOKEN.
