1 概述

本工具部署在您自己的 Cloudflare 账号下，通过读取您现有的 Logpush R2 日志，使用官方公式计算每分钟的回源带宽，并以图表形式展示。

项目说明
用途可视化您的源站需要向 Cloudflare 提供多少带宽（即缓存未命中流量）
技术栈100% Cloudflare 原生：Logpush + R2 + Queues + Workers + D1，零服务器维护
部署位置部署在您自己的 Cloudflare 账号下——无跨账号访问，无外部服务器
R2 访问方式通过 R2 Binding 只读访问，绝不向您的 R2 写入任何数据
数据保留7天（可配置），每日 UTC 02:00 自动清理
图表延迟约 20 分钟（安全窗口 + Cron 间隔）

使用的 Cloudflare 服务

服务职责Dashboard 位置
Logpush已在运行——每约1分钟将 HTTP 请求日志导出到 R2域名 → Analytics & Logs → Logpush
R2 对象存储存储 Logpush 文件。本工具通过 R2 Binding 只读这些文件，不写入Dashboard → R2
R2 Event Notifications每当 Logpush 写入新文件时，立即触发 Queue 消息——实现近实时处理R2 桶 → Settings → Event Notifications
Queues解耦 R2 事件检测与文件处理；提供至少一次投递保证和自动重试Dashboard → Workers & Pages → Queues
Workers一个脚本处理：Cron 扫描、Queue 消费（流式解析+写D1）、Dashboard HTTP通过 wrangler deploy 部署
D1（SQLite）存储分钟级带宽聚合数据，7天保留，自动清理Dashboard → Workers & Pages → D1

触发机制与延迟

阶段触发方式延迟
Logpush → R2时间驱动，约60秒间隔（CF 内部）整个链路唯一的"时钟"
R2 → Queue事件驱动：R2 Event Notification 在每次新文件写入时立即触发R2 写入后 2~5 秒
Queue → Worker事件驱动：满足 max_batch_size=3 或 max_batch_timeout=30s 时触发3~15 秒
Worker → D1在同一次 Worker 调用内完成毫秒级
链路总延迟Logpush(~60s) + Event(~5s) + Queue(~15s) + 处理(~30s)约 2 分钟
Safe Lag Window额外等待，确保该分钟所有文件都已落盘+10 分钟
图表数据延迟从请求发生到图表可见的总延迟约 12~15 分钟

关键设计决策

流式 Gzip 解压：Logpush 文件通过 DecompressionStream 从 R2 对象流中逐行解压处理，内存中始终只保存当前几行数据。无论文件多大（10 MB ~ 150 MB），内存占用保持恒定，不会触发 128 MB Worker 内存限制。

幂等累加写入：D1 写入使用 INSERT ... ON CONFLICT DO UPDATE SET sum_bytes = sum_bytes + excluded.sum_bytes。同一分钟内来自多个文件的数据会被正确累加。如果文件被重试，processed_files 状态检查会防止重复计算。

R2 Event Notification 为主要触发器：无需每 N 分钟轮询 R2，R2 Event Notification 实现近实时处理。Cron 触发器（每10分钟）仅作为兜底保障——应对偶尔遗漏的 Event Notification。

Safe Lag Window（安全延迟窗口）：跳过 now - LAG_MINUTES 之后写入的文件，确保某一分钟内的所有 Logpush 文件都已落盘再开始聚合，避免统计结果不完整。

3 计算公式

官方 Cloudflare 回源带宽计算公式：

SUM(CacheResponseBytes)
WHERE OriginResponseStatus NOT IN (0, 304)
÷ 60 秒 × 8 bits = Mbps

参考：developers.cloudflare.com/logs/faq/common-calculations

OriginResponseStatus含义是否计入带宽
0HIT — CF 从缓存返回，未联系源站❌ 排除
304Revalidated — 源站确认未变更，无 body 传输❌ 排除
200、206源站返回了内容（完整或分片），是回源带宽的主要来源✅ 计入
4xx、5xx源站错误响应（依官方公式，计入带宽）✅ 计入

时间分桶：使用 EdgeEndTimestamp（请求结束时间），与 CF Analytics 控制台统计口径一致。对于持续数秒甚至数分钟的大文件下载，按结束时间归入分钟更为准确。

4 部署步骤

前置条件：Node.js v18+，Wrangler CLI v3+（npm install -g wrangler），Cloudflare Workers Paid 计划（D1 和 Queues 需要付费计划），以及已配置并运行中的 Logpush 任务（正在向 R2 写入日志）。

步骤1 — 创建 D1 数据库

wrangler d1 create bw-stats

将输出中的 database_id 复制并填入 wrangler.toml：

[[d1_databases]]
binding       = "DB"
database_name = "bw-stats"
database_id   = "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"   # ← 粘贴到这里

步骤2 — 初始化 D1 表结构

wrangler d1 execute bw-stats --file=./schema.sql --remote

验证是否成功：

wrangler d1 execute bw-stats \
  --command="SELECT name FROM sqlite_master WHERE type='table'" --remote

应看到 bw_stats 和 processed_files 两张表。

步骤3 — 创建 Queue

wrangler queues create bw-ingest-queue
wrangler queues create bw-ingest-dlq

步骤4 — 修改 wrangler.toml

打开 wrangler.toml，填入您的实际配置：

[[r2_buckets]]
binding     = "LOGPUSH_BUCKET"
bucket_name = "your-logpush-bucket"   # ← 您的实际 R2 桶名

[[d1_databases]]
database_id = "xxxxxxxx-..."          # ← 步骤1中获取的 ID

[vars]
LOG_PREFIX = "logs/"                  # ← Logpush 写入路径前缀（通常是 "logs/"）
START_TIME = "2026-04-01T00:00:00Z"   # ← 只处理此时间点之后的文件（UTC）

步骤5 — 配置 R2 Event Notification（强烈推荐）

这是实现近实时处理的核心配置：

  1. CF Dashboard → R2 → 您的 Logpush 桶 → Settings → Event Notifications
  2. 点击 Add notification
  3. 设置：Event type = object-create，Queue = bw-ingest-queue
  4. 可选：Prefix = logs/，Suffix = .gz（只触发 Logpush 文件）
重要：不配置 R2 Event Notification 时，Worker 仅在每10分钟的 Cron 时触发。强烈建议配置 Event Notification 以实现及时处理。

步骤6 — 设置 Dashboard 鉴权 Token（可选）

# 留空 = 不鉴权（仅本地开发）
# 生产环境建议设置一个随机字符串：
wrangler secret put DASHBOARD_TOKEN

步骤7 — 部署

wrangler deploy

输出应显示：D1 binding、R2 binding、Queue 生产者/消费者、两个 Cron 调度。

步骤8 — 验证

# 查看系统状态
curl "https://origin-bw-monitor.<子域名>.workers.dev/api/status"

# 打开图表
# https://origin-bw-monitor.<子域名>.workers.dev/?zone=example.com&hours=24

5 配置说明

wrangler.toml [vars] 参数

变量名默认值是否必填说明
LOG_PREFIXlogs/必填R2 桶内 Logpush 文件的路径前缀，必须与您的 Logpush 任务配置一致。
START_TIME1970-01-01T00:00:00Z必填只处理此 UTC 时间点之后写入 R2 的文件。首次部署时建议设置为近期日期，避免处理数月的历史数据。
LAG_MINUTES10可选安全延迟窗口（分钟）。跳过 now - LAG_MINUTES 之后的文件，确保该分钟内所有 Logpush 文件已落盘。建议不低于 5 分钟。
SCAN_BATCH_SIZE20可选Cron 每次最多入队的文件数量。防止大量历史文件一次性涌入 Queue。手动导入历史数据请使用 /api/import。
MAX_PROCESSING_MIN15可选处理超时阈值（分钟）。如果一个文件在 processing 状态超过此时长，认为 Worker 已崩溃，文件将被重新入队。
DATA_RETENTION_DAYS7可选D1 数据保留天数。超过此天数的数据每日 UTC 02:00 自动删除。
LOG_LEVELinfo可选日志级别：debug / info / warn / error。排查问题时使用 debug，生产环境恢复为 info。
DASHBOARD_TOKEN空可选Dashboard 访问鉴权 Token。空 = 不鉴权（仅开发用）。生产环境通过 wrangler secret put DASHBOARD_TOKEN 设置。

6 历史数据导入

Cron 触发器负责自动处理新文件。如需导入历史文件（如过去30天的数据），使用 /api/import 接口，该接口通过 R2 Binding 直接读取 R2 文件。

基本导入（第一页）

curl -X POST "https://origin-bw-monitor.<子域名>.workers.dev/api/import" \
  -H "Content-Type: application/json" \
  -d '{
    "from_time": "2026-04-01T00:00:00Z",
    "page_size": 50
  }'

分页导入

如果响应中 "truncated": true，用返回的 next_cursor 继续：

curl -X POST ".../api/import" \
  -H "Content-Type: application/json" \
  -d '{
    "from_time": "2026-04-01T00:00:00Z",
    "page_size": 50,
    "cursor": "<上一次响应中的 next_cursor>"
  }'

重复执行直到 "truncated": false。建议每次请求间隔 30 秒以上，避免 Queue 过载。

查看导入进度

curl "https://origin-bw-monitor.<子域名>.workers.dev/api/status"

吞吐量估算：Worker 每秒处理约 1 个文件（受 R2 Binding 读取速度和 CPU 限制）。大批量历史导入（如5000个文件）约需 1~2 小时。
