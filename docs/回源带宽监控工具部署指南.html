1 概述

本工具部署在您自己的 Cloudflare 账号下，通过读取 Logpush 写入 R2 的日志文件，使用官方公式计算每分钟的回源带宽，并以实时图表形式展示，同时支持带宽超阈值和突增自动告警。

项目说明
用途可视化您的源站向 Cloudflare 提供的实际带宽（即缓存 MISS 产生的回源流量）
技术栈100% Cloudflare 原生：Logpush + R2 + Queues + Workers + D1，零服务器维护
部署位置部署在您自己的 Cloudflare 账号下——无跨账号访问，无外部服务器
R2 访问方式通过 R2 Binding 只读访问，绝不向您的 R2 写入任何数据
数据保留7天（可配置），每日 UTC 02:00 自动清理
图表延迟约 10~15 分钟（Safe Lag Window 7 分钟 + Cron 重试最多 8 分钟 + 处理时间约 1 分钟）

使用的 Cloudflare 服务

服务职责Dashboard 位置
Logpush已在运行——每约1分钟将 HTTP 请求日志导出到 R2域名 → Analytics & Logs → Logpush
R2 对象存储存储 Logpush 文件。本工具通过 R2 Binding 只读这些文件，不写入Dashboard → R2

Queues解耦 R2 事件与文件处理；提供至少一次投递保证和自动重试Dashboard → Workers & Pages → Queues

D1（SQLite）存储分钟级带宽聚合数据，7天保留，自动清理Dashboard → Workers & Pages → D1

触发机制与延迟

阶段触发方式延迟
Logpush → R2时间驱动，约60秒间隔（CF 内部）整个链路唯一的"时钟"
R2 → Queue事件驱动：R2 Event Notification 在每次新文件写入时立即触发R2 写入后 2~5 秒
Queue → Worker事件驱动：满足 max_batch_size=3 或 max_batch_timeout=30s 时触发3~15 秒
Worker → D1在同一次 Worker 调用内完成毫秒级
链路总延迟Logpush(~60s) + Event(~5s) + Queue(~15s) + 处理(~30s)约 2 分钟
Safe Lag Window等待该分钟所有文件全部落盘（7分钟），pending 文件由 Cron 每8分钟重试+7~15 分钟
图表数据延迟从请求发生到图表可见的总延迟约 10~15 分钟

关键设计决策

流式 Gzip 解压：Logpush 文件通过 DecompressionStream 逐行解压处理，内存恒定，不受文件大小影响，彻底避免 128 MB Worker 内存限制。

幂等累加写入：D1 写入使用 INSERT ... ON CONFLICT DO UPDATE SET sum_bytes = sum_bytes + excluded.sum_bytes。同一分钟来自多个文件的数据正确累加；文件重试也不会重复计算（processed_files 状态检查保护）。

Safe Lag Window + Cron 重试机制：新文件写入 R2 后立即被触发处理，但若文件中包含当前 lag 窗口内（7分钟）的日志行，则重置为 pending 状态，由 Cron 每8分钟重试，待 lag 窗口过后再完整处理。这保证了每一分钟的带宽数据的完整性。

3 计算公式

官方 Cloudflare 回源带宽计算公式：

SUM(CacheResponseBytes)
WHERE OriginResponseStatus NOT IN (0, 304)
÷ 60 秒 × 8 bits = Mbps

参考：developers.cloudflare.com/logs/faq/common-calculations

OriginResponseStatus含义是否计入带宽
0HIT — CF 从缓存返回，未联系源站❌ 排除
304Revalidated — 源站确认未变更，无 body 传输❌ 排除
200、206源站返回了内容（完整或分片），是回源带宽的主要来源✅ 计入
4xx、5xx源站错误响应（依官方公式，计入带宽）✅ 计入

时间分桶：使用 EdgeEndTimestamp（请求结束时间），与 CF Analytics 控制台统计口径一致。

4 部署步骤

前置条件：Node.js v18+，Wrangler CLI v3+（npm install -g wrangler），Cloudflare Workers Paid 计划（D1 和 Queues 需要付费计划），以及已配置并运行中的 Logpush 任务（正在向 R2 写入日志）。

步骤1 — 创建 D1 数据库

wrangler d1 create bw-stats

将输出中的 database_id 复制并填入 wrangler.toml：

[[d1_databases]]
binding       = "DB"
database_name = "bw-stats"
database_id   = "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"   # ← 粘贴到这里

步骤2 — 初始化 D1 表结构

wrangler d1 execute bw-stats --file=./schema.sql --remote

验证是否成功：

wrangler d1 execute bw-stats \
  --command="SELECT name FROM sqlite_master WHERE type='table'" --remote

应看到 bw_stats、processed_files、alert_history 三张表。

步骤3 — 创建 Queue

wrangler queues create bw-ingest-queue
wrangler queues create bw-ingest-dlq

步骤4 — 修改 wrangler.toml

打开 wrangler.toml，填入您的实际配置：

[[r2_buckets]]
binding     = "LOGPUSH_BUCKET"
bucket_name = "your-logpush-bucket"   # ← 您的实际 R2 桶名

[[d1_databases]]
database_id = "xxxxxxxx-..."          # ← 步骤1中获取的 ID

[vars]
LOG_PREFIX = "logs/"                  # ← Logpush 写入路径前缀（通常是 "logs/"）

步骤5 — 配置 R2 Event Notification（必须）

这是整个处理链路的触发源，必须配置，否则新文件写入 R2 后不会被自动处理：

  1. CF Dashboard → R2 → 您的 Logpush 桶 → Settings → Event Notifications
  2. 点击 Add notification
  3. 设置：Event type = object-create，Queue = bw-ingest-queue
  4. 可选：Prefix = logs/，Suffix = .gz（只触发 Logpush 文件，避免无关对象触发）
  5. 保存后，每当 Logpush 向 R2 写入新文件，Queue 即自动收到消息

步骤6 — 配置告警（可选，可完全跳过）

告警功能完全可选。如果您不需要告警，跳过本步骤即可——工具照常运行，只是不发送告警消息。

如需告警，在 wrangler.toml 中设置阈值，并至少配置一个 Webhook：

# wrangler.toml 中配置阈值（留空则不触发对应告警）
ALERT_THRESHOLD_MBPS   = "500"   # 超过 500 Mbps 触发阈值告警（留空=禁用）
ALERT_SPIKE_MULTIPLIER = "5"     # 超过过去24小时峰值5倍触发突增告警（留空=禁用）
ALERT_DASHBOARD_URL    = "https://origin-bw-monitor.your-subdomain.workers.dev"
# 配置 Webhook 接收端（支持任意组合，留空的渠道不会发送）
wrangler secret put ALERT_WEBHOOK_DINGTALK   # 钉钉（可选）
wrangler secret put ALERT_WEBHOOK_WECOM      # 企业微信（可选）
wrangler secret put ALERT_WEBHOOK_FEISHU     # 飞书（可选）

步骤7 — 设置 Dashboard 鉴权 Token（可选）

# 留空 = 不鉴权（仅本地开发）
# 生产环境建议设置：
wrangler secret put DASHBOARD_TOKEN

步骤8 — 部署

wrangler deploy

输出应显示：D1 binding、R2 binding、Queue Consumer、两个 Cron 调度（*/8 和 0 2）。

步骤9 — 验证

# 查看系统状态
curl "https://origin-bw-monitor.<子域名>.workers.dev/api/status"

# 打开图表（约10~15分钟后有数据）
https://origin-bw-monitor.<子域名>.workers.dev/?hours=24

5 配置说明

wrangler.toml [vars] 参数

变量名默认值是否必填说明
LOG_PREFIXlogs/必填R2 桶内 Logpush 文件的路径前缀，必须与您的 Logpush 任务配置一致。
LAG_MINUTES7可选Safe Lag Window（分钟）。跳过 now - LAG_MINUTES 之后写入的文件，等待该分钟内所有 Logpush 文件落盘后再聚合。建议不低于 5 分钟。
MAX_PROCESSING_MIN15可选处理超时阈值（分钟）。文件在 processing 状态超过此时长，认为 Worker 已崩溃，允许重试。
DATA_RETENTION_DAYS7可选D1 数据保留天数。超过此天数的数据每日 UTC 02:00 自动删除。
LOG_LEVELinfo可选日志级别：debug / info / warn / error。排查问题时使用 debug，生产环境恢复为 info。
ALERT_THRESHOLD_MBPS空（不启用）可选固定阈值告警（Mbps）。当前分钟带宽超过此值触发告警。示例："500"。
ALERT_SPIKE_MULTIPLIER空（不启用）可选突增倍数告警。当前带宽超过过去24小时最高值的 N 倍时触发。示例："5"。
ALERT_COOLDOWN_MIN30可选同 zone 同类告警的冷却时间（分钟），防止告警风暴。
ALERT_DASHBOARD_URL空可选告警消息中的面板跳转链接。填写后告警卡片底部显示"查看监控面板"按钮。
DASHBOARD_TOKEN空（不鉴权）可选Dashboard 访问鉴权 Token。生产环境通过 wrangler secret put DASHBOARD_TOKEN 设置。

GitHub Actions 自动部署

仓库包含 .github/workflows/deploy.yml，推送到 main 分支自动触发部署。需在 GitHub 仓库 Settings → Secrets → Actions 中设置以下4个 Secrets：

Secret 名称说明
CLOUDFLARE_API_TOKEN有 Workers Deploy 权限的 CF API Token
CLOUDFLARE_ACCOUNT_ID您的 Cloudflare Account ID
R2_BUCKET_NAMELogpush R2 桶名
D1_DATABASE_IDD1 数据库 ID（wrangler d1 create 输出）

实时查看 Worker 日志

wrangler tail origin-bw-monitor

正常日志示例：

[INFO] Processing: logs/20260414/20260414T100000Z_abc.log.gz
[INFO] Done: logs/20260414/... | lines=87234 lag=0 filter=83891 err=0 minutes=11
[WARN] Alert [threshold] zone=example.com mbps=623.5 threshold=500

清空数据重新统计

wrangler d1 execute bw-stats \
  --command="DELETE FROM bw_stats; DELETE FROM processed_files; DELETE FROM alert_history;" \
  --remote

清空后，新的 Logpush 文件写入 R2 时会自动重新触发处理。
